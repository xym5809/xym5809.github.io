同步脚本流程

合并执行项目中auto_merge.py代码即可。这里有补充的注释。请参考代码中TODO，修改路径等配置信息。

结果缓存目录是resultOutput。

伪代码

LAST_RELEASE_PATH = ''  # 上一个release版本的路径
CURRENT_RELEASE_PATH = ''  # 当前release版本的路径
PAD_CONFIG = {
    'project_path': '',  # 已拉取最新代码后的pad分支路径
    'diff_csv_name': ''  # pad分支差异对比文件列表
}
NOADS_CONFIG = {
    'project_path': '',  # 已拉取最新代码后的无广告分支路径
    'diff_csv_name': ''  # 无广告分支差异对比文件列表
}
IGNORE_LIST = [
    # 同步忽略文件，如build，manifest等，这些如果有涉及会体现在结果中的“not sure”中
]

if __name__ == '__main__':
    # 第一步，生成上一个版本的分支差异文件，避免上一个版本如果有新增差异文件，此版本没检测出来
    step1_cmp_dir(LAST_RELEASE_PATH, [PAD_CONFIG, NOADS_CONFIG])

    # 第二步，通过branch名称，或者通过时间，获取需要同步的文件的git日志
    gitlog = step2_git_log_by_branch(CURRENT_RELEASE_PATH, 'branch_include', 'branch_not_include', author='author 可选')
    # gitlog = step2_git_log_by_time（CURRENT_RELEASE_PATH,since,until,author='author 可选'）
    # 打印并检查git日志
    print_and_backup(gitlog, backup_path='备份文件路径 可选')

    gitlog = '''
    可以把第二步的git日志放在这里作为第三步的变量
    '''

    # 第三步，直接用第二步的日志变量，或者储存的文件路径，获取gitlog，来进行相关文件覆盖，到相应分支目录
    # 先选择要同步的是pad分支还是无广告分支，建议针对一个分支同时完成第三、第四步，再进行下一个分支
    config = PAD_CONFIG
    # config = NOADS_CONFIG
    step3_copy_file(gitlog, CURRENT_RELEASE_PATH, config, LAST_RELEASE_PATH,IGNORE_LIST)
    # step3_copy_file_by_path(git_log_file_path,CURRENT_RELEASE_PATH,config,LAST_RELEASE_PATH)
    # 说明：
    #  1.会生成同步结果进而备份文件如下：
    #    代码目录/backup/
    #     result+时间戳/（每次调用脚本都会生成一份）
    #       result.txt（合并结果）
    #         backup/（备份父目录）
    #           curent/(当前release分支)
    #           last/(上个release分支)
    #           before/(覆盖前的分支)
    #  对比建议用DiffMerge工具
    #  2.如果这步出错，请将要覆盖的分支先复原，不然会影响before的正确性。

    # 第四步，针对第三步的结果，对review文件和not sure文件进行手动同步，同步完成后，调用这个脚本来将手动同步结果也备份，方便他人查看和留档
    step4_backup_handled_result(config, backup_dir_path='')
    # 说明：
    # 1.backup_dir_path不填，会按照时间顺序将备份放到最近一次的备份父目录中（这是三、四步最好连续的原因），如果三、四步没有连续，则保证config中的路径是手动合并后的，backup_dir_path是相对应的“result+时间戳”目录即可。
    # 2.调用后，原本“result+时间戳”/backup/会新增一个目录handled，表示手动合并后的备份。

上述合并脚本是一次完整的合并流程，每次只进行一步操作。其他的步骤注释掉。

其中，第一步生成的“上一个版本的分支差异文件”，如果没问题，应该立即提交git。在同一个版本的合并中，第一步只需要进行一次。

第二步，补丁式的小提交，应当使用时间作为参数，无论是since还是until在统计gitlog时都没有被包含在内，所以请填写提交的前一天和后一天。另外注意获取到的gitlog应当检查一遍。毕竟时间作为参数获取到的结果可能是不准确的。

第三步，生成的结果文件的解释会后续补充到说明文档中。第三步的备份文件最重要的意义是，通过before和last的对比，来获取到文件被覆盖前，它和主分支的区别是什么？这样来给手动合并提供依据，避免手动合并误删了之前的区别。

注意，针对补充性的代码提交，第三步的备份文件是没有意义的。因为before和last的区别，已经包含了部分已完成的合并，所以无法区分出来。这种情况，建议直接看主分支都提交了哪些内容，以此为参考直接进行合并。同理，此时第4步的检验方式也不适用了。

第四步，生成的handled路径用来和current路径做一次对比，理论上，handled与current的差异 等同于 before和last的差异，这个差异即是这个分支与主分支的原本区别。此步骤非必须，只是用来验证第三步的结果是否正确。

关于第三步生成的结果文件解释

1.review 列表：

表示虽然覆盖了，但是这个文件之前就和主分支不一样。请查看有没有覆盖错误的、需要手动合并的部分。

2.error 列表：

包括modified error（未成功覆盖），added error（未成功添加），deleted error（未成功删除）。一般是由于git日志中前后矛盾导致，比如对于某个文件即有新增又有删除，这种情况按照删除进行了处理，那删除的时候肯定还没有添加，就没有执行。这种就需要具体文件具体分析。对照git日志看下前因后果即可。

3.成功列表：

包括modified、added、deleted，表示操作成功。

4.not sure列表：

分两种，1，是某些git日志，带R100，这个是按照rename来操作的，但是因为不确定是否是rename，所以提示出来。 2，是某些具体文件，是因为这个文件在IGNORE_LIST即忽略列表中，但是此次自动同步有所涉及，所以提示。
